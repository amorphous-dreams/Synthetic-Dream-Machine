# Prior Art: Transclusion and Inclusion Systems

Status: research pass on 2026-04-12.

This section drops the internal roleplay and collects concrete
systems with official documentation that can pressure-test the
memetic sigil model.

The goal is not "find things that vaguely resemble `<< >>`".
The goal is to find systems with:

- explicit inclusion or transclusion syntax
- documented processing model
- documented context or scope behavior
- documented failure modes
- enough structure to compare against `kahea`, `ahu`, wrappers,
  fragments, and active render handoffs

## Comparison Axes

Use these axes when comparing prior art to memetic sigils:

1. **Address form**
   direct name, path, URI, fragment, key, or query
2. **Inclusion type**
   static inclusion, parameterized inclusion, template masking,
   active transformation, or late-bound indirection
3. **Processing stage**
   preprocessor, parser, tree transformation, template runtime,
   or post-parse rendering
4. **Context behavior**
   whether the included content keeps its own context, inherits the
   caller's context, or receives an explicitly passed context
5. **Range/partial selection**
   whole object only, lines, tagged regions, fragments, element
   ranges, or filtered selections
6. **Error behavior**
   unresolved target, recursion loop, duplicate IDs, invalid scope,
   missing permissions
7. **Security / authority**
   whether inclusion is restricted by safe mode, namespace, map
   scope, or processor configuration

Why it matters:

- nearest live ancestor for the current memetic sigil direction
- strongest precedent for transclusion, templates, cascades,
  callable `<< >>` objects, and parser-driven lowering

Official docs:

- https://tiddlywiki.com/static/Transclusion.html
- https://tiddlywiki.com/static/Transclusion%2520in%2520WikiText.html

- parser modes and rule modules define the language operationally

## 2. MediaWiki

Why it matters:

- very mature transclusion culture

- https://www.mediawiki.org/wiki/Help:Extension:ParserFunctions

- `{{Page}}` transcludes a page by reference
- templates omit namespace in common cases
- parser functions use a visibly different invocation form:
  `{{#function: ... }}`
- substitution (`subst:`) exists as a one-time copy, not a live
  transclusion
- nesting depth and expensive function use are practical limits

What to test against memetic sigils:

- whether memetic sigils need an explicit "substitute once" mode
- whether active `kahea` should remain visibly distinct from static
  `kahea`, the way MediaWiki separates plain transclusion from
  parser functions
- parameter passing and recursion limits

Pressure point:

- MediaWiki's syntax can collapse many behaviors into one brace
  family; memetic sigils should stay more explicit than this, not
  less

## Working Synthesis

Across these systems, the reusable invariant pattern looks like
this:

1. a source object is addressed
2. the processor decides whether the act is:
   plain inclusion,
   parameterized inclusion,
   context-masked inclusion,
   or active transformation
3. the result is inserted or rendered under explicit scope rules
4. recursion, invalid targets, and security violations have
   documented behavior

That pattern supports the current memetic sigil split:

- `<<~kahea {meme-name} ... >>` maps well to direct inclusion /
  transclusion families
- `<<~kahea {meme-name} ... -> target >>` maps well to template,
  procedure, function, cascade, shortcode, and transform families

But the prior art also says something stricter:

- systems that survive at scale document the processing stage
- they document scope and context inheritance
- they document failure behavior
- they distinguish passive inclusion from active transformation

## Immediate Design Tests for Memetic Sigils

Before `v5`, the memetic sigil model should answer these directly:

1. Is bare `kahea` pre-parse inclusion, parse-time lowering, or
   runtime transclusion?
2. Does arrowed `kahea` receive caller context explicitly, or only
   by ambient inheritance?
3. Can `kahea` target partial body ranges, or only whole named
   memes and `ahu` worksites?
4. What is the exact behavior for unresolved source, unresolved
   target, recursion cycle, and permission attenuation failure?
5. Do we need a one-time substitution mode separate from live
   transclusion?
6. Do we need mode-like rendering, similar to XSLT modes or
   template views?
7. Do remote or cross-system calls require explicit opt-in, the
   way Asciidoctor gates URI includes?
