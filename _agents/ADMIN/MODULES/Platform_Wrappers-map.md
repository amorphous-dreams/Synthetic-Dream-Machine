## Overview

Three platform wrapper files append platform-specific coordinator suffixes to the combined output generated by `combine_agents.py`. Each wrapper is a **generated-file notice + worker registry table + platform-specific notes**. They are thin by design: the heavy payload currently remains `Lares_Preferences.md` + `Lares_VSCode_Operations.md`. Wrappers exist to isolate host-specific instructions without forking the core agent system.

The `platform/README.md` (362 lines) is the human-readable operations manual covering format specs, best practices, anti-patterns, and the Sprint 3 Codex implementation plan. It is **not** injected into any platform output — it is reference material only.

In the proposed modular architecture, wrappers should stay thin. They should evolve from "tail blocks appended to a monolith" into deterministic wrapper templates applied around a small root package plus scoped modules.

This makes the wrappers a **backlog contrast case**: they are not the main bug source, but they need reframing so they are clearly understood as host-integration artifacts inside the future deterministic build.

---

## Backlog Framing

**Current buggy state:** wrappers currently sit downstream of an oversized monolithic payload, which makes them look architecturally more important than they really are.

**Ideal state:** wrappers remain thin and explicit, applied as final host templates around resolved IaM packages built from manifests and module classes.

## Shared Structural Pattern

All three wrappers follow the same architecture:

1. generated-file/source-guard notice
2. worker registry block
3. platform-specific delegation/format notes
4. rebuild protocol

What changes across platforms:

- destination file path
- worker artifact format
- tool-name conventions
- invocation/delegation semantics
- size-budget and platform-limit notes

What should stay out of wrappers:

- large identity/lore sections
- repo source map material
- golden prompt examples
- regression test content
- generalized operating rules that belong in reusable modules

That boundary matters because wrappers should describe **host integration**, not carry shared runtime behavior.

## Current vs. Future Role

| Aspect | Current pipeline role | Future modular role |
|---|---|---|
| Core assembly | Appended after the same monolithic body on every platform | Wraps a target-specific manifest output |
| Shared behavior | Indirect only; most shared behavior lives elsewhere | Should contain none beyond wrapper-specific notes |
| Platform routing | Encodes worker format + host quirks | Same, plus points to scoped module strategy |
| Budget control | Mostly absent; wrappers only note constraints | Should participate in explicit budget/report checks |
| Build logic dependency | Marker-based extraction in `combine_agents.py` | Likely manifest/template driven, but still deterministic |

The wrappers are not the architectural problem. The problem sits in the identical 130+ KB core payload placed ahead of them. In the modular design, wrappers remain necessary, but they become the last few percent of the build rather than the only platform-specific distinction.

## Modular Extraction Notes

The three wrapper files are already appropriately thin. They do not need decomposition. The main modular risks are:

1. **Wrapper drift** — registry tables or rebuild notes diverge across platforms unintentionally.
2. **Host-note leakage** — platform-specific caveats expand until wrappers start duplicating shared operational content.
3. **Marker fragility** — the combine script currently relies on one heading marker per wrapper file.

Recommended future handling:

- keep wrapper files as source-of-truth notes for host quirks
- move shared worker registry data into one reusable source if drift appears
- convert wrapper application from marker slicing to explicit templates/manifests when the build system becomes manifest-driven

The Sprint 3 plan proposes a trimmed Codex-variant build path in `combine_agents.py` that selectively excludes lararium archaeology, degraded-node names, and exchange vector theory when building root `AGENTS.md`. Even if that happens as an interim fix, wrappers should remain thin and template-like.

The `## [Platform] Platform — Worker Registry` marker in each wrapper is what `combine_agents.py` uses to locate and append the platform suffix. Do not rename these markers without updating the combine script.
