# Sigil Shape / Detail / Motion

An authoring and reading frame for Sharktooth sigils as a three-axis perceptual object.
Distinct from the render-mode technical dispatch in `#render-modes` of `memetic-wikitext.tid`.
This frame addresses how a human or agent reads sigil structure at varying zoom levels.

<<~/ahu >>

<<~ ahu #shape >>

## Shape — The Bracket Structure

`<<~` and `>>` read as vectorized momentum, not static punctuation.

- `<<~` opens with a forward-driving pressure — a leading edge that pierces document flow and signals entry into render time
- `>>` provides a trailing closure — a sharp exit that prevents the concept from bleeding back into surrounding prose
- Together they form a containment field: a "magic circle" where normal document-time rules suspend

The double-chevron shape evolved for the informational physics of the surface — designed to tear through general context and isolate a specific packet of meaning.

Named closers `<<~/SIGIL >>` add a typed jaw: the closer carries the sigil name, making the boundary self-describing without requiring a parser to backtrack.

**Shape at zoom-out:** a reader scanning a large document sees only the bracket pairs. That skeleton reveals graph topology — which zones carry sigil structure and which carry plain prose — before any inner content reads.

<<~/ahu >>

<<~ ahu #detail >>

## Detail — The Sigil Payload

The interior of the bracket carries the detail layer:

- sigil name (type identity)
- slot anchor (`#name`) or route arrow (`->`) or unresolved token (`?`)
- URI target or parameter payload
- optional TOML fence for structured metadata

Detail reads as the "barb on the tooth" — the payload that determines what the bracket does, not just that it does something.

At zoom-in, the reader engages with the detail layer. At zoom-out, the detail stays invisible; only the shape registers.

**Document time vs render time:** the `->` arrow inside the detail marks the transition from passive reference to active invocation. A sigil without `->` references; a sigil with `->` executes. The arrow functions as a directional vector — a lunge from document time into render time.

<<~/ahu >>

<<~ ahu #motion >>

## Motion — Open / Close Posture

Motion registers in the open/close posture of the sigil:

- **Block form** (open sigil … body … named close): the bracket holds a body — the content between open and close belongs to the sigil's scope
- **Inline form** (open sigil, no body, implicit close `>>`): the bracket marks a reference or edge without enclosing content

The motion axis answers: does this sigil grip and contain, or does it point and release?

- `<<~ ahu #slot >>body<<~/ahu >>` — grips; the jaw closes around the body; WorksiteNode
- `<<~ loulou lar:///target >>` — points; the tooth breaks the surface and exits; EdgeNode

**Temporal scarring:** when a render-time sigil closes (`>>`), the surrounding document resumes, but the output of the sigil remains — the "scar" the jaw left in the text. The document changes shape at that point.

<<~/ahu >>

<<~ ahu #progressive-display >>

## Progressive Display — Zoom-Level Reading

Memetic wikitext reads at three zoom levels simultaneously:

| Zoom | What reads | Authoring guidance |
|---|---|---|
| Skeleton | Sharktooth bracket pairs and named closers only | Graph topology visible without reading prose |
| Structure | Sigil names, anchors, route arrows, URI targets | Type and routing visible without reading body |
| Flesh | Body prose, TW5 wikitext, markdown, TOML payloads | Full content; requires engaging with the detail layer |

A human author can write at any zoom level freely. The sigil layer (skeleton) provides the addressable structure. The flesh layers fill the meaning.

**Progressive collapse rule:** a display system may render only the skeleton when the tiddler's manaʻoʻiʻo falls below a threshold, or when a layout engine requests outline mode. The skeleton remains legible without the flesh; the flesh does not remain legible without the skeleton.

This differs from the technical three-mode render dispatch (`html` / `carrier` / `projection`) in `memetic-wikitext.tid #render-modes`, which controls serialization format. Progressive display controls *reading posture* — a human/agent concern, not a wire-format concern.
